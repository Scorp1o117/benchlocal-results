⚡ 关于 BenchLocal

BenchLocal 是一个本地优先的桌面应用，用于运行、比较和管理可安装的 LLM Bench Packs。支持本地或远程模型，通过可安装的测试包对模型进行标准化评估。

📋 测试项目

  • ToolCall-15 — 15 题工具调用测试，覆盖参数提取、多轮上下文、并行调用等
  • BugFind-15 — 15 题跨语言代码调试，含 Trap 陷阱题，难度 Easy~Expert
  • HermesAgent-20 — 20 题 Agent 场景测试，覆盖记忆管理、技能创建、调度投递等
  • 能力上限 — 加权原始分 = ToolCall×0.3 + BugFind×0.3 + HermesAgent×0.4（模型最高能拿多少分）
  • 实用得分 — 能力上限 − 重试扣分（每题首次通过不扣分，重试后通过每重试1次扣1分，重试后仍失败不参与扣分）
  • 排序方式 — 能力上限排序（看天花板）、实用得分排序（看实际表现）、三项专项排序

📊 测试总结

基于 BenchLocal 在同一硬件环境下的本地推理测试，横向对比 11 个模型配置

🏆 核心发现

  • 能力上限 vs 实用得分差距显著 — 重试次数越多，两者的差距越大：N2-Mini 差 27 分（87.2→60.2），27B 无思考版仅差 3 分（87.4→84.4）
  • ToolCall 满分常见 — Step-3.7、QwenPaw-9B、27B 思考版、DeepSeek 均满分
  • API 大模型作为参考 — DeepSeek-V4-Flash（284B MoE）能力上限 94.0，实用得分 85.0，三项均衡
  • BugFind BF-03 Trap 题全场最难 — 多数模型在"代码没bug"场景翻车，无法判断代码是正确的

⚠️ 已知局限

  • Trap 题普遍失分 — BF-03（代码没bug）和 BF-10（红鲱鱼）是多数模型的软肋
  • HA-07（代码批量处理）大部分失败 — execute_code 能力普遍薄弱
  • HA-16（消息投递）全军覆没 — 所有模型都无法找到正确的消息通道
  • 重试成本差异巨大 — 最少 3 次（27B 无思考）vs 最多 27 次（N2-Mini），稳定性差距明显

💡 测试环境

  • 硬件 — RTX 5070 Ti 16GB + 128GB RAM，MoE模型部分专家层offload到CPU
  • 推理后端 — llama.cpp
  • 测试包 — ToolCall-15 / BugFind-15 / HermesAgent-20（共 50 题）
